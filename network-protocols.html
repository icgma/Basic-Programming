网络协议基础入门指南

了解 HTTP、HTTPS 以及网络通信的基础概念。对于编写爬虫和理解 API 调用至关重要。

1. HTTP 协议 (超文本传输协议)

HTTP (HyperText Transfer Protocol) 是万维网数据通信的基础。它是一个客户端和服务器端之间请求和应答的标准。

主要特点：
  • 无连接：每次连接只处理一个请求。服务器处理完客户的请求，并收到客户的应答后，即断开连接。
  • 无状态：协议对于事务处理没有记忆能力。后续处理需要前面的信息，则它必须重传。
  • 明文传输：数据在传输过程中没有加密，存在被窃听的风险。

2. HTTPS 协议 (安全超文本传输协议)

HTTPS (HyperText Transfer Protocol Secure) 是以安全为目标的 HTTP 通道，在 HTTP 的基础上通过传输加密和身份认证保证了传输过程的安全性。

与 HTTP 的区别：
  • HTTPS 需要到 CA 申请证书，一般免费证书较少，因而需要一定费用。
  • HTTP 是超文本传输协议，信息是明文传输，HTTPS 则是具有安全性的 SSL 加密传输协议。
  • HTTP 和 HTTPS 使用的是完全不同的连接方式，用的端口也不一样，前者是 80，后者是 443。

3. 常见的 HTTP 请求方法

在网络爬虫和 API 调用中，最常用的是 GET 和 POST 请求。

  • GET：向指定的资源发出“显示”请求。使用 GET 方法应该只用在读取数据，而不应当被用于产生“副作用”的操作中（例如修改数据）。
  • POST：向指定资源提交数据，请求服务器进行处理（例如提交表单或者上传文件）。数据被包含在请求本文中。
  • PUT：向指定资源位置上传其最新内容。
  • DELETE：请求服务器删除 Request-URI 所标识的资源。

4. HTTP 状态码

状态码由三位数字组成，第一个数字定义了响应的类别。

  • 2xx (成功)：请求已成功被服务器接收、理解、并接受。如：200 OK。
  • 3xx (重定向)：需要客户端采取进一步的操作才能完成请求。如：301 Moved Permanently。
  • 4xx (客户端错误)：客户端看起来可能发生了错误，妨碍了服务器的处理。如：404 Not Found。
  • 5xx (服务器错误)：服务器在处理请求的过程中有错误或者异常状态发生。如：500 Internal Server Error。

5. URL 结构剖析 (统一资源定位符)

了解 URL 的组成部分，有助于在爬虫中精确构造请求目标和传递参数。

标准格式：scheme://host:port/path?query#fragment
  • Scheme (协议)：指明如何访问资源，如 http 或 https。
  • Host (主机)：存放资源的服务器域名系统 (DNS) 主机名或 IP 地址。
  • Path (路径)：由零或多个“/”符号隔开的字符串，用来表示主机上的一个目录或文件地址。
  • Query (查询字符串)：通过 GET 方式请求时传递的参数，通常以 ? 开始，键值对用 & 分隔（如 ?page=2&sort=desc）。

6. 核心请求头 (HTTP Headers)

请求头包含了关于客户端环境和请求正文的元数据。在编写爬虫时，合理伪装请求头是突破反爬策略的第一步。

  • User-Agent：告诉服务器客户端的操作系统、浏览器版本等信息。如果不带此头，Python 的 Requests 库会默认发送 python-requests/x.x，极易被识别为爬虫并拦截。
  • Cookie：存储在客户端的凭证数据。很多网站（如微博、知乎）需要登录后才能查看完整数据，此时就需要在请求头中携带 Cookie。
  • Referer：告诉服务器该请求是从哪个页面链接过来的。部分图片或视频资源通过检查 Referer 来实现防盗链。
  • Accept：声明客户端可以处理的媒体类型，如 application/json 或 text/html。

7. 常见数据交互格式

当你向服务器发出请求后，服务器通常会以这两种主要格式返回数据。

  • HTML (超文本标记语言)：你在浏览器中直接看到的网页结构。爬取此类数据需要使用 BeautifulSoup 或 lxml 进行节点树解析。
  • JSON (JavaScript 对象简谱)：一种轻量级的数据交换格式，易于人阅读和编写，也易于机器解析。现代网站的数据大多通过 API 接口以 JSON 格式异步加载。对于爬虫而言，抓取 JSON 接口是最优解，可以直接将其转换为 Python 字典（response.json()）。

8. RESTful API 设计理念

目前互联网上绝大多数公开的数据接口（如 GitHub API, Twitter API）都遵循 RESTful 风格。

核心特征：
  • 资源驱动：将网络上的每一个实体（如一篇新闻、一个用户）都看作一个资源，用 URL 来定位。
  • 动词对应方法：使用不同的 HTTP 方法（GET、POST、PUT、DELETE）来代表对资源的不同操作（查、增、改、删）。例如 GET /users 是获取用户列表，而 POST /users 是创建新用户。
  • 无状态性：每次请求都必须包含服务器处理该请求所需的所有信息（例如把鉴权 Token 放在 Header 中），服务器不在 Session 中保存客户端的上下文状态。
